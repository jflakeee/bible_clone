## 1. 테스트 전략 개요

### 1.1 테스트 피라미드

```
        /  E2E  \           ← 10% (Playwright)
       /----------\
      / 통합 테스트 \        ← 30% (Jest + Supertest)
     /--------------\
    /  단위 테스트    \      ← 60% (Jest + React Testing Library)
   /------------------\
```

- **단위 테스트 (60%):** 개별 컴포넌트, 서비스, 유틸리티 함수의 독립적 검증
- **통합 테스트 (30%):** API 엔드포인트, 데이터베이스 연동, 서비스 간 상호작용 검증
- **E2E 테스트 (10%):** 사용자 시나리오 기반 전체 흐름 검증

### 1.2 테스트 도구

| 도구 | 용도 | 적용 범위 |
|------|------|----------|
| Jest | 단위/통합 테스트 | 프론트엔드 + 백엔드 |
| React Testing Library | 컴포넌트 테스트 | Next.js 프론트엔드 |
| Playwright | E2E 테스트 | 크로스 브라우저 시나리오 |
| Supertest | API 통합 테스트 | NestJS 엔드포인트 |
| k6 | 성능/부하 테스트 | API 및 전체 시스템 |
| Lighthouse | 성능/접근성 측정 | 프론트엔드 |
| OWASP ZAP | 보안 테스트 | 전체 시스템 |
| axe-core | 접근성 테스트 | 프론트엔드 컴포넌트 |
| pgTAP | DB 테스트 | PostgreSQL 함수/프로시저 |

### 1.3 테스트 환경

| 환경 | 용도 | 데이터베이스 | 비고 |
|------|------|-------------|------|
| 로컬 개발 | 단위/통합 테스트 | SQLite 또는 Docker PostgreSQL | 개발자 로컬 환경 |
| CI/CD (GitHub Actions) | 자동화 테스트 | Docker PostgreSQL | PR 머지 전 필수 통과 |
| 스테이징 | E2E/성능 테스트 | Supabase (스테이징) | 운영과 동일 구성 |
| 운영 | 스모크 테스트 | Supabase (운영) | 배포 후 핵심 기능 검증 |

#### 2.1.3 E2E 테스트

- [ ] 개역한글에서 KJV로 번역본 전환 후 동일 위치 유지 확인
- [ ] KJV에서 WEB으로 번역본 전환 후 동일 위치 유지 확인

- [ ] URL 직접 입력으로 특정 절에 접근 가능한지 확인 (딥 링크)
- [ ] 모바일 뷰포트에서 텍스트 뷰어 레이아웃 정상 표시 확인

#### 2.1.4 엣지 케이스

- [ ] 가장 긴 장(시편 119편, 176절) 로딩 및 렌더링 성능 확인
- [ ] 가장 짧은 장(요한3서 등 단일 장 책) 네비게이션 처리 확인
- [ ] 성경의 첫 절(창세기 1:1)에서 이전 절 이동 시도 시 처리
- [ ] 성경의 마지막 절(요한계시록 22:21)에서 다음 절 이동 시도 시 처리
- [ ] 특수 문자 포함된 절 텍스트 렌더링 (셀라, 히브리어 알파벳 제목 등)
- [ ] 네트워크 끊김 상태에서 캐시된 데이터 표시 동작 확인
- [ ] 동시에 여러 장을 빠르게 전환할 때 레이스 컨디션 방지 확인
- [ ] 존재하지 않는 성경 참조 URL 접근 시 적절한 에러 페이지 표시

#### 2.1.5 성능 테스트

- [ ] 성경 한 장 로딩 시간 < 500ms (API 응답 기준)
- [ ] 첫 화면 로딩 시간 (LCP) < 2.5초
- [ ] 장 전환 시 UI 업데이트 시간 < 200ms
- [ ] 동시 100명 사용자 접속 시 API 응답 시간 < 1초

